CEO Command CenterRoadmap a Producto Final + Checklist reunión con ingeniero de Odoo

Autor: Klaus Uribe  ·  Fecha: 16 de abril de 2026  ·  Versión: 1.0  ·  Estado actual: Fase 0 completada, backend FastAPI corriendo junto a Streamlit

1. Contexto y estado actual

El CEO Command Center corre hoy como una aplicación Streamlit funcional (MVP) con datos demo sintéticos sobre SQLite. La arquitectura de tres capas ya separa analítica (analytics/), motor de IA (ai/) y UI, lo cual permite introducir un backend FastAPI sin reescribir lógica.

Ya entregado (antes de este roadmap):

  • 10 páginas Streamlit funcionales con KPIs, charts, tablas y análisis IA por módulo.
  • Módulo de Presupuesto con proyección histórica y supuestos estructurados.
  • Autenticación vía streamlit-authenticator + bcrypt.
  • 78 tests de analytics/KPIs + 13 tests de backend = 91 verdes.
  • Fase 0 del backend: FastAPI con JWT, 5 endpoints (/health, /auth/login, /auth/token, /auth/refresh, /auth/me, /kpis/all), OpenAPI automática, Docker Compose con Postgres, convive con Streamlit sin romper nada.

2. Roadmap completo a producto final

Quedan 6 fases para llegar a "entrega final para cliente". Tiempo total estimado: 7-10 semanas de trabajo efectivo solo-dev.

# Fase Entregables clave Tiempo
1 API Layer completo Todas las funciones de analytics/ y ai/ expuestas como endpoints REST tipados, organizados por módulo: /api/sales/*, /api/receivables/*, /api/payables/*, /api/inventory/*, /api/expenses/*, /api/financial/*, /api/cashflow/*, /api/budget/*, /api/ai/*. Schemas Pydantic estrictos por módulo. Paginación y filtros. Rate limiting básico. 1-2 sem
2 Frontend base Next.js 15 (App Router) + TypeScript + Tailwind v4 + shadcn/ui scaffolding. NextAuth v5 para sesiones. Layout con sidebar. Cliente API auto-generado desde OpenAPI (orval). Primera página end-to-end: Resumen Ejecutivo con KPIs y charts ECharts. 1 sem
3 Migración de páginas Las 10 páginas Streamlit recreadas en Next.js: Ventas, CxC, CxP, Inventarios, Gastos, Financiero, Flujo de Caja, Presupuesto, AI Chat, Importar Datos. Stack de cada página: TanStack Query para fetching, ECharts para visualización, TanStack Table v8 para tablas con virtualización, React Hook Form + zod para formularios. 2-3 sem
4 ETL real de Odoo XML-RPC client de producción con pool de conexiones, retry + backoff exponencial, delta sync por write_date, paginación de resultados. Sync jobs por módulo con APScheduler. Reemplaza scripts/generate_demo_data.py como fuente de verdad. Logging estructurado en sync_log. 1-2 sem
5 Producción Migración SQLite → Postgres con Alembic. Rate limiting (slowapi). Sentry para errores. Security headers (CSP, HSTS, X-Frame-Options). Logging estructurado a Axiom/Logtail. Dockerfile multi-stage optimizado. CI/CD con GitHub Actions. Deploy: Vercel (frontend) + Fly.io (backend) + Neon/Fly Postgres. Custom domain + HTTPS automático. 1-2 sem
6 QA + Docs + Handoff Suite E2E completa con Playwright. Load testing básico (K6). Auditoría de accesibilidad. Manual de usuario en español (PDF imprimible). Guía de administrador (backup, restore, gestión de usuarios). Runbook de incidentes. API docs publicadas (Redocly). 3-5 días
Tip de paralelización: Fases 1 y 2 pueden arrancar en paralelo si alguien colabora con frontend. Fase 4 (Odoo ETL) es independiente del frontend — podés construirla mientras se migran páginas. Fase 6 siempre va al final.
Decisión de gating: Fase 4 depende críticamente de las respuestas del ingeniero de Odoo. Sin esas definiciones, se puede hacer el ETL pero con suposiciones que quizá haya que deshacer. Por eso esta reunión es el mayor desbloqueador del proyecto.

3. Checklist para reunión con ingeniero de Odoo

Tiempo estimado: 45-60 min. Los ítems marcados con ⚠ son decisiones que bloquean Fase 4.

Bloque A — Stack y configuración

  • Versión exacta de Odoo (v16 / v17 / v18) y Community vs Enterprise
  • Hosting: ¿Odoo.sh, servidor propio, on-prem?
  • Lista de módulos instalados relevantes: sale, account, stock, purchase, l10n_bo, account_budget, hr_payroll

Bloque B — Contabilidad (bloque más crítico)

  • ¿Usan contabilidad analítica (analytic_distribution)? Si no, ¿se puede activar?
  • Si usan: ¿qué cuentas analíticas corresponden a cada centro de costo del dashboard (Admin, Comercial, Marketing, Almacén, RRHH, TI)?
  • Plan de cuentas localizado Bolivia (l10n_bo) o custom? Pedir CSV export de account.account con columnas: code, name, account_type, group_id
  • ¿Cómo están agrupadas las cuentas (account.group) en: Ingresos, Costos, Gastos Operativos, Gastos Financieros, Impuestos, Activo Corriente, Activo No Corriente, Pasivo Corriente, Pasivo No Corriente, Patrimonio?
  • ¿Manejan presupuestos en Odoo? ¿Con qué módulo (account_budget, custom)?

Bloque C — Productos y clientes

  • ¿Cómo clasifican productos por línea/marca/segmento? (product.tag de Odoo 17+, categorías jerárquicas, custom fields, nada)
  • ¿Clasificación A/B/C de clientes existe en Odoo? ¿Dónde?
  • ¿credit_limit y payment_terms están completos en todos los clientes?

Bloque D — Ventas

  • ¿Usan sale.order (proceso completo) o solo account.move (facturación directa)?
  • ¿Vendedor se registra en sale.order.user_id, account.move.invoice_user_id, o custom field?
  • ¿Existen metas por vendedor en algún lado? (crm.team, spreadsheet aparte, nada)

Bloque E — Inventario y compras

  • ¿Cuántos stock.warehouse? ¿Reportamos unificado o separado?
  • ¿Usan stock.warehouse.orderpoint (reordenamiento automático)?
  • Valuación: standard_price, promedio, FIFO?
  • ¿Cuentas por pagar tienen alguna clasificación de prioridad? (o lo calculamos nosotros)

Bloque F — Cash Flow

  • ¿Cómo clasifican hoy movimientos en Operativo / Inversión / Financiamiento? (reporte nativo de Enterprise, criterio por cuenta, a mano)
  • ¿Todos los cobros/pagos van por account.payment o hay asientos manuales directos?

Bloque G — Acceso técnico (crítico para el ETL)

  • ¿Pueden crear un usuario técnico read-only con permisos a: account.move, account.move.line, res.partner, product.*, stock.quant, stock.move, account.account, account.analytic.*, account.payment, crm.team, res.users?
  • URL de Odoo para XML-RPC (https://xxx.odoo.com o IP interna/VPN)
  • ¿API keys habilitadas (Odoo 14+) o solo user/password?
  • ¿Rate limiting? ¿Cuántas llamadas/min soportan sin bloquearme?
  • Ventana horaria para correr sync pesado (nocturno)

Bloque H — Volumetría y datos

  • ¿Cuántos años de histórico tienen?
  • Facturas/día aproximadas (para estimar carga del ETL)
  • ¿Hay data en Excel u otro ERP que no está en Odoo y que deberíamos ingestar?
Tip: pedir 3 CSVs de muestra. Antes de cerrar la reunión, pedirle que exporte 10 filas de cada uno:
  1. account.move — últimos 30 días, filtrado a facturas de venta (move_type='out_invoice')
  2. res.partner — 10 clientes activos (customer_rank > 0)
  3. stock.quant — 10 productos con stock en locaciones internas
Con esos CSVs se valida el mapping antes de escribir una sola línea de ETL. Si aparecen campos faltantes, ahorra 1-2 semanas de rework.
Bandera roja: Si el ingeniero dice "no usan contabilidad analítica", el módulo de centros de costo del dashboard se queda sin fuente de datos. Soluciones posibles: (a) activar analítica ahora y backfillear, (b) mapear por cuenta de gasto, (c) sacar los centros de costo del dashboard. Decidir en la reunión.

4. Los 6 gaps del audit traducidos a decisiones

Estas son las decisiones concretas que deben quedar cerradas en la reunión o poco después. Cada gap tiene 2-3 opciones viables; la columna Recomendación es lo que yo sugiero basado en tamaño de la empresa y esfuerzo de implementación.

Gap Pregunta concreta y opciones Recomendación
1. Product line / brand
(Ventas – Pareto, rankings)		 ¿Cómo clasifican productos por línea y marca?
  • A. product.tag en Odoo 17+ (dos tags: línea + marca)
  • B. Custom fields x_product_line, x_brand en product.template
  • C. Sacar esos breakdowns del dashboard
 Opción A si están en v17+; sino Opción B con un par de campos
2. Seller targets
(Ventas – ranking de vendedores)		 ¿Existen metas mensuales por vendedor? crm.team.invoiced_target es por equipo, no por persona.
  • A. Tabla local editable desde nuestra UI (como hicimos con Presupuesto)
  • B. Custom field x_target_monthly en res.users
  • C. Solo tracking por equipo — pierde ranking individual
 Opción A — simple, desacopla del ciclo de release de Odoo
3. Payables priority
(CxP – plan de pagos)		 La columna priority (critical / high / normal / low) no existe nativa.
  • A. Reglas automáticas: critical si days_overdue > 30 AND amount > X, etc.
  • B. Custom field x_priority en account.move
  • C. Combinación A + override manual local para casos especiales
 Opción C — reglas automáticas + override manual
4. Expense category
(Gastos – fijos/variables)		 Clasificación fixed / variable / semi-variable.
  • A. Mapping por cuenta contable en config (5100 → variable, 6200 → fixed)
  • B. Custom field x_expense_category en account.account
  • C. Heurística por estabilidad (variación mes-a-mes < 5% → fixed)
 Opción A — se hace una vez, sobrevive cambios en Odoo
5. Cost centers
(Gastos – por centro de costo)		 Depende 100% de contabilidad analítica en Odoo (account.analytic.account).
  • A. Activar analítica en Odoo y asignar analytic_distribution en todos los asientos
  • B. Dejar el módulo vacío hasta que la activen
  • C. Mapeo por cuenta contable hacia un centro de costo (menos granular)
 Opción A es la única profesional — empujar para activarla
6. Cashflow classification
(Flujo – op/inv/fin)		 La clasificación Operativo / Inversión / Financiamiento no es automática.
  • A. Reporte nativo account.cash.flow.report (solo Enterprise)
  • B. Mapping por rango de código de cuenta (5xxx, 6xxx = operativo; 1200-1299 = inversión)
  • C. Custom field x_cashflow_category en account.account
 Opción B — rápido y no depende de Enterprise
Resumen de lo que tiene que salir de la reunión: versión de Odoo confirmada, decisión sí/no sobre contabilidad analítica, credenciales de un user read-only, URL del XML-RPC, y las 6 decisiones de la tabla de arriba. Todo eso cabe en una hoja A4 de notas.

5. Tips y comentarios finales

Para la reunión con el ingeniero

Postura recomendada: tratá la reunión como un discovery técnico, no como un pedido. El ingeniero está viendo esto por primera vez y puede decir "no" por defecto a cosas como activar contabilidad analítica. Mostrale el dashboard corriendo (Streamlit en :8501) para que entienda el valor de activar esas piezas.
  • No comprometerse con deadlines de ETL hasta tener respuestas en mano.
  • Grabá la reunión (con permiso) — vas a agradecer poder volver a los detalles técnicos.
  • Si el ingeniero ofrece crear custom fields, aceptá solo si están versionados en un módulo custom (no directamente vía UI) — si no, se pierden en upgrades.
  • Pedí la lista de usuarios/equipos de ventas para poder mapear seller_id correctamente desde el día uno.

Para la ejecución del roadmap

  • No skipear Fase 1. Es tentador ir directo al frontend, pero sin API completa vas a estar parchando endpoints mientras migrás páginas.
  • Migrar páginas en orden de valor, no alfabético: Home → Ventas → CxC → Flujo de Caja → Financiero → Gastos → CxP → Inventarios → Presupuesto → Chat IA. Las primeras 4 son las que el CEO abre todos los días.
  • Deploy temprano, deploy seguido. Desde Fase 2 en adelante, cada merge a main debería triggerear un deploy a staging automático. Feedback real > review en local.
  • Mantener Streamlit vivo hasta que la última página esté en Next.js. El equipo de negocio puede seguir usando la vieja mientras migramos.

Para el posicionamiento como producto

Desde Fase 5 (producción), este proyecto deja de ser "dashboard interno" y empieza a ser portfolio piece de AI Solutions Builder. Sugerencias:
  • Grabá un walkthrough en video de 3-5 min para mostrar a clientes.
  • Escribí un case study (Medium / LinkedIn) explicando la arquitectura: Streamlit MVP → FastAPI + Next.js producción → ETL Odoo. Ese es el arco narrativo que vende.
  • Extraé las partes genéricas (motor de análisis IA, engine de cache, forecaster) como un "boilerplate" reusable para futuros clientes.

Riesgos y contingencias

RiesgoImpactoMitigación
Odoo en v14 o menor (sin product.tag, sin API keys) Alto Fallback a custom fields en product.template y user/password auth
Sin contabilidad analítica y cliente no quiere activarla Alto Eliminar vista por centro de costo o derivarla por cuenta contable
Volumetría mayor a la esperada (>10k facturas/día) Medio Delta sync desde el día 1, nunca full refresh de account.move
XML-RPC lento o con rate limit agresivo Medio Usar search_read con limit + paginación + queue con retry exponencial
